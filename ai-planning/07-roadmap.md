# 07 — Roadmap, Milestones & Open Questions

## Phasing

### Phase 0 — Project skeleton ✅
- Vite + React + TypeScript (strict) scaffold.
- GitHub Pages deploy via GitHub Actions; correct `base` / hash routing.
- Hash routing, app shell, empty screens (Test / Results / History).
- **Exit criteria:** a blank app deploys live to GitHub Pages.

### Phase 1a — Minimal working test loop ✅
- 1 hard-coded passage (Bunny and Duck, ~132 words, Easy tier).
- Lexical matching engine: tokenise → normalise (basic) → compare → completion predicate. 42 unit tests green.
- Test screen: passage display, Start button, dictation textarea, auto‑complete trigger.
- Results screen: WPM + CPM + elapsed time via router state.
- **Exit criteria:** you can complete a dictation test and see your speed.

### Phase 1b — Persistence ✅
- `localStorage` schema (`sst.v1`) + history module — save, load, clear, storage-availability check (see [06](./06-history-and-storage.md)).
- Paste detection: results flagged `suspect` when match jumps from <5 tokens to complete.
- Save result before navigating to results screen.
- History screen: reverse-chronological table, two-step clear, storage-unavailable banner.
- **Exit criteria:** results survive page refresh.

### Phase 1c — Matching polish ✅
- Strict mode (case-sensitive, no punctuation stripping).
- Sentence-terminator rules: statement (`.`/`!` interchangeable) vs question (`?` distinct).
- Abbreviation/decimal exclusions so `Dr.` and `3.14` are not sentence breaks.
- Number canonicalisation (custom implementation — no third-party library). Handles 0–9,999; ordinals keep suffix (`"third"` ≡ `"3rd"`); greedy parse for compounds.
- Live progress highlighting: correct prefix highlighted, first mismatch token flagged.
- 120 unit tests green (42 engine + 78 new, covering terminators, abbreviations, ordinals, hundreds/thousands, sentence boundaries).
- **Known limitation:** `refTokens.length` may differ from `rawTokens.length` when number canonicalization collapses multi-word spans (e.g. `"twenty five"` → `"25"`). Live highlighting indices are reliable only for passages without spelled-out numbers. Passages must be audited before commit (Phase 1f).
- **Exit criteria:** both modes correct; UI shows live progress through the passage.

### Phase 1d — Corpus classification tooling ✅
- `scripts/classify-passage.ts`: given passage text, loads `data/google-10000-english-no-swears.txt`, computes the tier-composition vector `(p₁, p₂, p₃, p₄)`, difficulty bin, and a punctuation-complexity report (flags problematic constructs — dialogue quotes, em-dashes, nested punctuation — that could interfere with STT matching).
- `scripts/build-frequency.ts`: `google-10000-english-no-swears.txt` → `data/frequency.json` + `data/frequency-meta.json` (word → tier map, pinned commit recorded).
- Tier lookup module `src/corpus/tiers.ts` for use by both the script and the browser.
- `tsconfig.scripts.json` + `npm run typecheck:scripts` added; scripts type-check cleanly.
- **Empirical baseline:** bunny-and-duck passage → vector [0.326, 0.265, 0.295, 0.114], difficulty *medium*. Inflected verb forms (hopped, listened, smiled) are absent from the surface-form frequency list, structurally inflating p4 for all narrative prose. Provisional thresholds: easy ≤ 0.08, medium ≤ 0.14 (calibrate in Phase 1f).
- **NOTE:** `data/frequency-meta.json` has `"sourceCommit": null` — fill in the git hash from `first20hours/google-10000-english` that was vendored.
- **Exit criteria:** running `classify-passage.ts` on any text file prints its composition vector, difficulty bin, word count, and a list of punctuation warnings. ✅

### Phase 1e — Frequency pipeline wired into the app
- `data/frequency.json` + `data/frequency-meta.json` generated by the Phase 1d build script and committed.
- `data/passages.json` schema established; hard-coded passages migrated into it with precomputed composition vectors.
- Difficulty bin shown in results screen; optional filter by difficulty on test screen.
- **Exit criteria:** frequency data is live in the browser and difficulty is a first-class concept.

### Phase 1f — Passage creation (incremental)
- Passages are added **one at a time**, each authored to a specific difficulty target (Easy / Medium / Hard).
- Each passage is run through `scripts/classify-passage.ts`; punctuation warnings are resolved before committing.
- Classified passage is appended to `data/passages.json`.
- Build up to ~30–60 passages spanning the full difficulty range.
- **Exit criteria:** corpus is large and varied enough for meaningful testing.

### Phase 2 — Real WPM (Method A) + history viz
- Difficulty‑binned harmonic estimate (Method A in [05](./05-realwpm.md)).
- "Your Real WPM" panel with per‑tier breakdown + confidence flags.
- Trend chart, personal bests, export/import.
- **Exit criteria:** a meaningful, caveated Real WPM appears after a handful of tests.

### Phase 3 — Gutenberg corpus pipeline
- Offline `scripts/ingest-gutenberg.ts`: acquire → strip boilerplate → segment → filter → classify → emit.
- Expand corpus to the bounded cap (≤ ~1,000 passages), difficulty‑balanced, lazy‑loaded shards.
- Attribution UI for sourced passages.
- **Exit criteria:** corpus is mostly book‑derived and difficulty‑balanced.

### Phase 4 — Real WPM (Method B) + polish
- Constrained inverse‑WPM **NNLS** regression, regularised toward Method A.
- Confidence bands; "take more Hard passages" nudges.
- Strict/Lexical parity, accuracy/cleanliness stats, accessibility pass.

## Open questions (need a decision; defaults proposed)

| # | Question | Proposed default |
|---|----------|------------------|
| Q1 | Frequency list source? | **Decided:** `google-10000-english-no-swears.txt` (first20hours), vendored at a pinned commit. Web‑derived; tier = line number. |
| Q2 | Are `?` and `!` strict like full stops, or ignorable? | **Decided:** `.` and `!` are one interchangeable *statement* class; `?` is a distinct *question* class that must match. Every sentence end requires an appropriate terminator. |
| Q3 | Clock starts on Start click or first matching token? | **Decided: on Start click.** Many STT engines batch‑emit all text at the end; first‑match timing would measure flush latency, not human speaking time. |
| Q4 | Display character metric: raw or normalised? | **Raw** (store both). |
| Q5 | Lemmatise before tier lookup? | **No** — surface forms only, keeps matching & tiering identical. |
| Q6 | Number canonicalisation edge cases (`"twenty twenty"`, ordinals, currency)? | Greedy digit parse; ordinals keep suffix; currency **not** canonicalised v1. |
| Q7 | Styling: Tailwind vs CSS Modules? | **Decided:** `styled-components`. Co-located component styles, no class-name management, TypeScript-friendly theming. |
| Q8 | Passage selection: pure random or difficulty‑targeted? | Random by default; allow filtering by bin/length. |

## Risks

- **Real‑WPM identifiability** (collinear compositions) — mitigated by binning + tier‑skewed passages + confidence flags; cannot be fully eliminated. *Set expectations in UI.*
- **Number/abbreviation normalisation correctness** — biggest source of "why won't it complete?" frustration; mitigated by showing the expected normalised token on mismatch and heavy unit tests.
- **STT variance across tools** — different engines punctuate/casing differently; Lexical mode + consistent timing reduce, don't remove, this.
- **Bundle size** from corpus — bounded + lazy‑loaded shards.
- **Gutenberg licensing/attribution** — ship derived passages + attribution, not whole books; verify terms.

## Definition of done (v1)

A deployed GitHub Pages site where a user can: pick/receive a passage, dictate it via their own STT, auto‑complete, and get WPM/CPM; see a history of results in `localStorage`; and view a caveated "Your Real WPM" with a per‑tier breakdown — all with no backend.
